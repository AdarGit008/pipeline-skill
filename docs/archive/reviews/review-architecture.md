## Note (precision gaps, risks, follow-ups — none block the build)

1. **BR-03's comparand is unspecified — "the CI pipeline's check name(s)".** (RULES.md:62) GitHub check names are `<workflow name> / <job name>`; with no YAML parser, deriving them by line-scan is fragile and the derivation rule is never stated. Two builders will pick different comparand sets for a rule that escalates to blocker at critical. *Smallest fix:* add a config key in ARCHITECTURE §8 (e.g. `required_check_names: []`; empty = derive from `name:` lines of CI-01-matched workflows, labeled heuristic) and reference it in BR-03's CHECK.
2. **CI-02's regex under-verifies the locked claim.** (RULES.md:101) The alternation `npm test|pytest|…|tox|build` ends in a bare `build`, so a workflow running only `docker build` / `npm run build` — zero tests — PASSes a **blocker** titled "builds & tests on PR/push" whose own SOURCE requires "each commit … trigger a series of automated tests". R1 locked certainty by scoping the claim to presence, but as written the presence claim itself is satisfied by build-alone. *Smallest fix:* drop the bare `build` alternative (build is implied by every test invocation) or require a build-signal AND a test-signal; otherwise re-scope the title/rationale to "builds on integration triggers".
3. **ENV-01's PASS condition is ambiguous.** (RULES.md:140) "PASS when all consulted sets agree (one-element tolerance: a tier present only in workflows is a finding, not a pass)" — "one-element tolerance" and the parenthetical pull in opposite directions; the set algebra is unspecified. *Smallest fix:* state it exactly, e.g. "PASS iff descriptor-set == forge-set == workflow-set; forge unreachable → descriptor-set == workflow-set; any symmetric difference FAILs naming the diff."
4. **DESC-03's fire condition embeds undefined judgment.** (RULES.md:37) "Fires when the observed tier-set materially exceeds the declared tier" — "materially" is doing all the work. Labeled `heuristic` (so honest), but the tier-artifact mapping is the rule's entire content and is undefined. *Smallest fix:* define the mapping once (e.g. gated prod env or ≥2 forge envs ⇒ critical-level artifacts; staging-tier workflow ref or IaC presence ⇒ team-level) and the fire rule (observed level > declared level ⇒ one WARN).
5. **HOT-04's forge cross-ref names an endpoint that may not exist.** (RULES.md:312) `gh api repos/{owner}/{repo}/rulesets/.../insights` — GitHub Rule Insights is a UI surface; no public REST endpoint for it is documented, and the audit-log alternative is org/enterprise-admin scoped. The design is safe (verdict never depends on it; degrades to labeled `not consulted`), but a builder will burn time hunting the endpoint. *Smallest fix:* add one clause — "endpoint availability unverified; implement behind a capability probe, absence is always `forge: not consulted`" — or demote the cross-ref to a documented manual step in REFERENCE.md.
6. **Forge-plan gating is never mentioned.** BR-02 (required reviews) and ENV-04 (environment protection rules) require paid-plan features on private repos; a free-plan private repo at team+/critical cannot PASS them and gets no hint why. The deviation ledger covers it, but only if the user understands the failure. *Smallest fix:* one sentence in RULES.md's forge-convention preamble ("How to read this catalog") noting plan-gated forge features resolve via `deviation` when the plan can't provide them.
7. **IAC-02's non-Terraform arm is a shrug.** (RULES.md:193) "evaluates via tool-appropriate markers" — markers not enumerated (Pulumi: `backend:` key in Pulumi.yaml; CDK: tool-managed, arguably SKIP). *Smallest fix:* name the markers per tool, or declare non-Terraform tools always SKIP-with-detail.
8. **FLAG-01's "source globs" are never enumerated** (RULES.md:234) and no `source_globs` key exists in ARCHITECTURE §8's config. *Smallest fix:* enumerate defaults in the CHECK (e.g. `src/**/*.{js,ts,py,go,rb}` + dependency manifests) or add the config key.
9. **FLAG-02 vacuous-pass hole.** (RULES.md FLAG-02 CHECK) Opt-in active (`uses_feature_flags: true`) with zero `records/flags/*.json` passes "every record has an owner" vacuously — the registry the family exists to police is never required. *Smallest fix:* state the empty-registry behavior (recommend WARN-detail "flag registry empty"; check baseline's claims-field zero-records behavior at build time and mirror it).
10. **RB-02's recommended tripwire needs computed comparands.** (RULES.md:266) `today gt <last-drill + rollback_drill_days>` — baseline's tripwire ops compare a fact against a literal (verified src/jdg.mjs:107-119); `gt` on two ISO-date strings works, but the comparand must be precomputed at authoring time. "Recommended", so non-blocking. *Smallest fix:* one clause in ARCHITECTURE §6.1 stating `jdg new` computes date-valued tripwire comparands when authoring.
11. **SPEC §5's "Both READMEs say so" over-promises cross-repo.** (SPEC.md:88) The baseline reference repo's README contains no pipeline mention (verified by grep) and lives outside this build's scope. *Smallest fix:* reword to "pipeline's README says so; a sibling note in baseline's README is a follow-up" or track the baseline edit as an explicit follow-up task.
12. **Minor unused machinery.** `json-field` is listed among reused kinds (ARCHITECTURE §5.1) but no locked rule uses it — harmless (unknown-kind self-check only fails on *unknown* kinds), but a builder may wonder. Either note "implemented for completeness" or drop it from the list.
