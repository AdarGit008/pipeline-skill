# pipeline-skill — Rule Catalog (LOCKED v0.1)

**Status: LOCKED.** 39 rules in 9 families. IDs are frozen — never renumber; a retired rule is tombstoned, not reused. Every rule below satisfies the hard constraint: *no rule rests on a written promise alone.* Each is either (a) mechanically checkable by a zero-dependency Node script on a repo **at rest** (tree = file presence/content, history = local git, forge = `gh` API, ledger = judgment records) or (b) an explicit, dated, expiring **sign-off ledger entry**.

## How to read this catalog

- **Severity** — `blocker` (FAIL sets exit 1) · `warning` (advisory; engine value `warn`) · `sign-off` (satisfied only by a dated, unexpired `kind: sign-off` judgment whose subject is the rule id; engine value `manual`).
- **Profiles** — cumulative failure-cost tiers: `all` = active at solo+team+critical · `team+` = team and critical · `critical` = critical only. `opt-in` marks the FLAG family: it activates only when the descriptor sets `uses_feature_flags: true` or `records/flags/` exists (baseline's claims opt-in pattern).
- **Certainty** — `deterministic` (pure presence/absence or set-comparison claim) · `heuristic` (labeled line-scan inference, never dressed up as proof) · `judgment` (a script cannot decide; ledger only).
- **Locked engine laws** (inherited from baseline's `--self-check`): **a blocker must be deterministic; a sign-off must be judgment.** Both hold across every profile escalation in this catalog.
- **CHECK** describes the mechanical check precisely (files/globs/workflow line-scans/git or `gh` queries) or names the ledger entry. **FIX** is the concrete remediation the `fix` mode applies. **SOURCE** is a verbatim quote + URL copied from the research briefs in `docs/archive/research/` — nothing here is cited from memory. Rules marked *machinery* police the skill's own mechanism (descriptor, ledger); their precedent is baseline-skill itself, stated explicitly rather than dressed in an external citation.
- **`applies_to`** draws from the closed, deployment-shaped set `service | app | library | docs | infra` (decision D3). `infra` = a repo whose primary content is IaC.
- **Plan-gated forge features** — required reviews (BR-02), environment protection rules (ENV-04), and ruleset/protection reads (BR-01/03) are paid-plan features on private repos. When the repo's plan cannot provide the mechanism, the FAIL resolves as a dated `deviation` judgment naming the plan limit — never a silent exception.

## DESC — descriptor & declared posture

The single declaration point the whole taxonomy polices: `pipeline.repo.json` at the repo root (schema: `schema/repo.schema.json`). Fields: `type`, `profile` (`solo|team|critical`), `branching_model` (`trunk|dev-staging-main|gitflow`), `default_branch`, `environments[]` (name, tier, gated), `iac` (tool, state backend), `uses_feature_flags`, `branches` (optional map — `integration`/`staging`/`production` name the tier branches; unset keys fall back to the `branching_model` conventions: `trunk` → the default branch, `gitflow` → `develop`, `dev-staging-main` → `dev`/`staging`/`default_branch`), `recovery_objective`.

### DESC-01 — `pipeline.repo.json` descriptor exists
- **Severity:** warning · **Profiles:** all · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** Profile, branching model, environments, and recovery objective cannot be reliably inferred from the tree; a minimal descriptor is the one stored intent every cross-check (DESC-03, BR-04, ENV-01, RB-04) hangs off. Absence is invited-not-forced (warning); invalidity is DESC-02's blocker.
- **CHECK:** `descriptor` — presence of `pipeline.repo.json` at the repo root (tree).
- **FIX:** Copy the closest `config-presets/*.repo.json` preset (or `templates/pipeline.repo.json`) to the repo root and set `type`, `profile`, `branching_model`, `default_branch`, `environments`.
- **SOURCE:** "Environments are used to describe a general deployment target like `production`, `staging`, or `development`." — GitHub Docs, *Deployment environments* — https://docs.github.com/en/actions/concepts/workflows-and-actions/deployment-environments (the canonical tier vocabulary the descriptor's `environments[]` enumerates; from `docs/archive/research/cicd-environments.md`, Core claim C). Machinery precedent: baseline `rules/desc.json` DESC-01.

### DESC-02 — A present descriptor is schema-valid
- **Severity:** blocker · **Profiles:** all · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** Every profile gate and cross-check consumes the descriptor; an invalid one silently demotes the whole taxonomy to SKIPs while the run stays green-looking. Deterministic, so blocker is lawful: present-but-invalid fails loudly. Absence stays DESC-01's warning — presence and content are different findings.
- **CHECK:** `descriptor-valid` — hand-rolled JSON-Schema validation (zero-dependency, as baseline `src/validate.mjs`) of `pipeline.repo.json` against `schema/repo.schema.json`, including `profile ∈ solo|team|critical` and `branching_model ∈ trunk|dev-staging-main|gitflow` (tree).
- **FIX:** Fix the named schema errors, or re-copy a `config-presets/*.repo.json` preset and re-fill the identity fields.
- **SOURCE:** machinery rule — no external claim; mechanism precedent: baseline-skill `rules/desc.json` DESC-02 (presence/content severity split).

### DESC-03 — Declared profile matches observed complexity (the inverse-nag)
- **Severity:** warning · **Profiles:** all · **Certainty:** heuristic · **applies_to:** all
- **Rationale:** The philosophy made structural: complexity you declared you don't need (staging+prod+gated promotion under `solo`) is overhead to stop paying; a declaration that outruns reality (`critical` with no prod gate) is posture you don't carry. Either direction fires exactly one warning; a legitimate mismatch resolves as a dated `deviation` judgment, never a silent exception.
- **CHECK:** `profile-consistency` (new kind) — compare descriptor `profile` against observed profile-gated artifacts: count of forge environments (`gh api repos/{owner}/{repo}/environments`), gated-promotion signals (ENV-04's read), staging-tier workflow references, IaC presence. Observed level mapping: a gated prod environment **or** ≥2 forge environments ⇒ critical-level artifacts; a staging-tier workflow reference, IaC presence, or branch-protection/review signals ⇒ team-level; none of these ⇒ solo-level. Fires (exactly one WARN) when the observed level exceeds the declared tier, or the declared tier is critical and no prod-gating artifact is found (tree + forge; forge unreachable → tree-only comparison, labeled).
- **FIX:** Reconcile declaration and reality: right-size the descriptor, remove the unjustified tier, or record `pipeline jdg new --kind deviation --subject DESC-03 --reason "why the mismatch is deliberate" --review-by <date>`.
- **SOURCE:** "Traditionally, these goals have been met through a heavyweight process involving approval by people external to the team proposing the change: a change advisory board (CAB) or a senior manager. However, DORA's research shows that these approaches have a negative impact on software delivery performance. Further, no evidence was found to support the hypothesis that a more formal, external review process was associated with lower change fail rates." — DORA, *Capabilities: Streamlining change approval* — https://dora.dev/capabilities/streamlining-change-approval/ (from `docs/archive/research/flags-rollback.md` §5.1; anchors the "don't carry unjustified gating" direction. The tiering itself is the skill's design judgment — see SPEC.md §2).

## BR — branching & PR workflow

### BR-01 — Default branch is protected
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** Green CI means nothing if anyone can force-push or commit directly to the default branch; protection is a forge fact, so the rule asks the forge rather than trusting a committed description of it.
- **CHECK:** `forge-protection` (reused kind, `gov: protection`) — read branch protection/rulesets for the descriptor's `default_branch` via `gh api`; PASS when force-push and direct-push are disallowed (forge; `on_unreachable: skip` — offline, enforcement is honestly unknowable and the SKIP is labeled).
- **FIX:** Add a ruleset/protected-branch rule on the default branch disallowing force-push and requiring PR flow: `gh api repos/{owner}/{repo}/rulesets --input <ruleset.json>`.
- **SOURCE:** "After enabling required status checks, all required status checks must pass before collaborators can merge changes into the protected branch." — GitHub Docs, *About protected branches* — https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches (from `docs/archive/research/cicd-environments.md`, Claim 1).

### BR-02 — PR review is required
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** Review is the team's knowledge-distribution mechanism and the lightweight gate DORA's evidence supports (in contrast to external heavyweight approval); the requirement must live in forge settings, not in a contributing doc.
- **CHECK:** `forge-protection` extension (`gov: reviews`) — read required approving review count on the default branch; PASS when count ≥ 1 (forge; `on_unreachable: skip`).
- **FIX:** Set required approving reviews ≥ 1 in the branch ruleset/protection settings.
- **SOURCE:** "Teams with faster code reviews have 50% higher software delivery performance." — DORA, *2023 Accelerate State of DevOps Report* — https://dora.dev/research/2023/dora-report/2023-dora-accelerate-state-of-devops-report.pdf (**honesty label:** quote-verified via search snippet, full PDF not re-fetched — adequate as supporting evidence for a warning; re-fetch the PDF if this citation ever becomes load-bearing — `docs/archive/research/cicd-environments.md`, Core claim F, gap 3) · The rationale's heavyweight-approval contrast: "Further, no evidence was found to support the hypothesis that a more formal, external review process was associated with lower change fail rates." — DORA, *Capabilities: Streamlining change approval* — https://dora.dev/capabilities/streamlining-change-approval/ (added at review: the contrast now rests on BR-02's own SOURCE, not only DESC-03's).

### BR-03 — Required status checks gate the default branch
- **Severity:** warning at team · **blocker at critical** (`severity_by_profile`) · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** The practice claim survived research as written: every change built and tested, failing pipeline blocks merge. The check verifies the *enforcement mechanism* both forges document — required checks on the integration branch, in strict/up-to-date mode so a stale branch can't merge green.
- **CHECK:** `forge-protection` extension (`gov: required-checks`) — required status checks on the default branch include the pipeline's check name(s) — from config `required_check_names` when set, else derived from the `name:` lines of the CI-01-matched workflow files (labeled heuristic derivation: GitHub check names are `<workflow name> / <job name>`, and a line-scan can't always prove them) — and strict/up-to-date policy is on (forge; `on_unreachable: skip` — a critical-tier repo scored offline gets a labeled SKIP, never a silent PASS).
- **FIX:** In the ruleset, require the pipeline's status checks and enable "require branches to be up to date before merging".
- **SOURCE:** "After enabling required status checks, all required status checks must pass before collaborators can merge changes into the protected branch." — and, on strict mode: "**Strict** ... The branch **must** be up to date with the base branch before merging. This is the default behavior for required status checks." — GitHub Docs, *About protected branches* — https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches · Also: "You can configure your project to require a complete and successful pipeline before merge." — GitLab Docs, *Auto-merge* — https://docs.gitlab.com/user/project/merge_requests/auto_merge/ (both from `docs/archive/research/cicd-environments.md`, Claim 1).

### BR-04 — Declared branching model matches observed branches
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** The descriptor declares a model (`trunk` / `dev-staging-main` / `gitflow`); the remote either carries that shape or the declaration is fiction. The rule flags the *mismatch*, not the model — no primary source condemns GitFlow-style models, and none is enforced here.
- **CHECK:** `branch-model` (new kind) — descriptor `branching_model` vs `git ls-remote --heads`: `trunk` expects no long-lived secondary integration branches (`develop`, `staging`, `release/*`); `dev-staging-main` expects the tier branches present (descriptor `branches.integration`/`branches.staging`, falling back to `dev`/`staging` when unset); `gitflow` expects `develop` + default branch (history/forge — `ls-remote` degrades to local refs when offline, labeled).
- **FIX:** Align the remote with the declaration (create/prune tier branches) or correct `branching_model` in the descriptor.
- **SOURCE:** "Analysis of DORA data from 2016 (PDF) and 2017 (PDF) shows that teams achieve higher levels of software delivery and operational performance (delivery speed, stability, and availability) if they follow these practices: * Have three or fewer active branches in the application's code repository. * Merge branches to trunk at least once a day. * Don't have code freezes and don't have integration phases." — DORA, *Capabilities: Trunk-based development* — https://dora.dev/capabilities/trunk-based-development/ (from `docs/archive/research/cicd-environments.md`, Core claim A; evidence is associational at team level — hence warning, not blocker).

### BR-05 — No stale unmerged long-lived branches
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** all
- **Rationale:** Stale unmerged branches are batch-size inflation: they concentrate merge risk and quietly become the release train nobody rides. The rule flags *staleness*, never the branch model itself; a justified long-lived branch is a dated `risk-acceptance`.
- **CHECK:** `stale-branches` (new kind) — unmerged branch tips (not ancestors of the default branch) whose tip commit is older than config `stale_branch_days` (default 30); protected tier branches declared by `branching_model` are exempt (history; purely local git).
- **FIX:** Merge, rebase, or delete the named branches; for a deliberately long-lived branch, `pipeline jdg new --kind risk-acceptance --subject BR-05 --reason "..." --review-by <date>`.
- **SOURCE:** "branches in trunk-based development typically last no more than a few hours, with many developers merging their individual changes into trunk frequently." — vs. long-lived branches: "These changes require bigger and more complex merge events when compared to trunk-based development. This approach also requires additional stabilizing efforts and 'code lock' or 'code freeze' periods to make sure the software stays in a working state, because large merges frequently introduce bugs or regressions." — DORA, *Capabilities: Trunk-based development* — https://dora.dev/capabilities/trunk-based-development/ (from `docs/archive/research/cicd-environments.md`, Core claim A).

### BR-06 — Release points are marked
- **Severity:** warning · **Profiles:** all · **Certainty:** deterministic · **applies_to:** all
- **Rationale:** A marked production version is what a hotfix branches from and what "redeploy previous" points at; without tags or releases there is no shared name for "what's live".
- **CHECK:** `any-of` over: tags present (`git tag -l`, history) · forge releases endpoint non-empty (`gh api repos/{owner}/{repo}/releases`, forge) · a release workflow file exists (tree glob `.github/workflows/*release*`). Repos younger than config `release_mark_grace_days` (default 90, from first commit date) return not-evaluable → SKIP: a trunk shop that hasn't cut its first release is never wallpapered.
- **FIX:** Tag the current production ref (`git tag v0.1.0 <sha> && git push --tags`) or cut a forge release; wire a release workflow if releases are pipeline-made.
- **SOURCE:** "When a critical bug in a production version must be resolved immediately, a hotfix branch may be branched off from the corresponding tag on the master branch that marks the production version." — Vincent Driessen (nvie), *A successful Git branching model* — https://nvie.com/posts/a-successful-git-branching-model/ (from `docs/archive/research/hotfix-incidents.md`, D1; the tag-as-production-marker practice BR-06 exists to guarantee. **Source class:** nvie is the canonical-origin primary source — the model's original description; official-vendor corroboration for the GitFlow family lives in GitLab Docs *Branching strategies*, cited in HOT-01/HOT-02).

## CI — CI/CD pipeline stages

### CI-01 — A CI pipeline definition exists
- **Severity:** blocker · **Profiles:** all · **Certainty:** deterministic · **applies_to:** `["service","app","library","infra"]`
- **Rationale:** Every other delivery rule presupposes an automated pipeline; "we run checks by hand" is the failure mode the entire standard exists to retire. Pure presence/absence over known CI paths — deterministic.
- **CHECK:** `any-file` over `.github/workflows/*.yml`, `.github/workflows/*.yaml`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/config.yml`, `azure-pipelines.yml` (tree).
- **FIX:** Add a CI definition (e.g. `.github/workflows/ci.yml`) that builds and tests on push/PR.
- **SOURCE:** "A CI system that runs the build and automated tests on every check-in." — DORA, *Capabilities: Continuous integration* — https://dora.dev/capabilities/continuous-integration/ (from `docs/archive/research/cicd-environments.md`, Claim 1).

### CI-02 — The pipeline builds & tests on PR/push
- **Severity:** blocker · **Profiles:** all · **Certainty:** deterministic · **applies_to:** `["service","app","library","infra"]`
- **Rationale:** The claim is scoped precisely to what a line-scan proves: *your checked-in pipeline definition declares build+test on integration triggers.* Whether those checks gate the merge is BR-03's forge-side job; whether the tests are good is baseline's TEST family. A repo whose CI lives outside the scanned files (external SaaS with no repo file) false-negatives in one direction only, and resolves honestly via `risk-acceptance` — never silently. (Certainty lock: see §Locked resolutions, R1.)
- **CHECK:** `file-contains`-style line scan over the CI-01 glob set: a workflow file must contain an integration trigger (`pull_request`, `push`, `merge_request`) AND a build/test invocation pattern (`npm (run )?test|pytest|go test|cargo test|make test|gradle (test|check)|mvn (test|verify)|tox` step regex — a bare `build` deliberately absent: a build-only workflow runs zero tests, and the locked claim is build **and** test) (tree).
- **FIX:** Add trigger keys and a test step to the workflow; external/unreadable CI → `pipeline jdg new --kind risk-acceptance --subject CI-02 --reason "CI is hosted at <system>, definition not in-repo" --review-by <date>`.
- **SOURCE:** "The key elements in successfully implementing continuous integration are: * Each commit should trigger a build of the software. * Each commit should trigger a series of automated tests that provide feedback in a few minutes." — and: "The 2015 State of DevOps Report (PDF) shows that teams perform better when developers merge their work into trunk at least daily. A set of automated tests is run both before and after the merge in order to validate that the changes don't introduce regression bugs. If these automated tests fail, the team stops what they are doing to fix the problem immediately." — DORA, *Capabilities: Continuous integration* — https://dora.dev/capabilities/continuous-integration/ (from `docs/archive/research/cicd-environments.md`, Claim 1 — **survives as written**).

### CI-03 — Deployment is automated
- **Severity:** blocker · **Profiles:** all · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Deploy-by-hand is the named failure source, and the citation holds with no team-size precondition — so this blocks even at solo. Scoped claim: *a deploy step/workflow is declared in the checked-in pipeline definitions.* A genuinely manual-by-design deploy is a documented `deviation`, flagged precisely because primary sources treat console-deploying as a defect, not an end state.
- **CHECK:** line scan over CI workflow files for deploy signals: `environment:` key, deploy keywords (`deploy`, `aws ... deploy`, `flyctl deploy`, `vercel`, `netlify deploy`, `kubectl apply`, `terraform apply`, `serverless deploy`, `pulumi up`, `cdk deploy`), or a dedicated `*deploy*` workflow file (tree). A library repo (`type: library`) SKIPs as `n/a` — publishing is not deployment.
- **FIX:** Add a deploy job/workflow (promote the same artifact the pipeline built); manual-by-design → `pipeline jdg new --kind deviation --subject CI-03 --reason "..." --review-by <date>`.
- **SOURCE:** "Deployment automation is what enables you to deploy your software to testing and production environments with the push of a button. Automation is essential to reduce the risk of production deployments." — and: "Use the same deployment process for every environment, including production." — and: "Any deployment process that requires logging into a console and interacting manually by clicking around should be a target for improvement." — DORA, *Capabilities: Deployment automation* — https://dora.dev/capabilities/deployment-automation/ · Also: "Releases are truly automatic, and only require engineer involvement if and when problems arise." — Google, *Site Reliability Engineering*, ch.8 — https://sre.google/sre-book/release-engineering/ (from `docs/archive/research/cicd-environments.md`, Claim 2 — **survives as written**, holds at solo).

### CI-04 — A post-deploy smoke test runs in the deploy path
- **Severity:** warning · **Profiles:** critical · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** DORA defines the deployment pipeline as *including* a deployment test — the deploy path is incomplete without it. Presence-of-mechanism check only; smoke-test *adequacy* folds into ENV-06's sign-off residue.
- **CHECK:** line scan of deploy workflow(s): a smoke/health-check step pattern (`smoke`, `health`, `curl ... /health`, `wait-on`, `check` job) appearing after a deploy step in the same file (tree; heuristic — no YAML parser, step ordering inferred by line position).
- **FIX:** Add a smoke step to the deploy workflow that hits the service's health endpoint and fails the job on non-2xx.
- **SOURCE:** "An automated deployment process has the following inputs: ... Scripts to configure the environment, deploy the packages, and perform a deployment test (sometimes known as a *smoke test*)." — and, step 5 of the deploy script: "Perform a deployment test to make sure that any necessary external services are reachable, and that the system is functioning." — DORA, *Capabilities: Deployment automation* — https://dora.dev/capabilities/deployment-automation/ · Also: "*Smoke tests*, in which engineers test very simple but critical behavior, are among the simplest type of system tests. Smoke tests are also known as *sanity testing*, and serve to short-circuit additional and more expensive testing." — Google, *Site Reliability Engineering*, ch.17 — https://sre.google/sre-book/testing-reliability/ (from `docs/archive/research/cicd-environments.md`, Claim 5 — **survives as written**).

### CI-05 — The pipeline is alive at the forge
- **Severity:** warning · **Profiles:** all · **Certainty:** deterministic · **applies_to:** `["service","app","library","infra"]`
- **Rationale:** A scheduled/triggered workflow the forge has auto-disabled files nothing forever while every dashboard stays green — the death mode is silence, so liveness is a checked fact, not an assumption. Lifts baseline OPS-07's `workflow-state` pattern, pointed at the delivery workflows. Scope note: GitHub's 60-day inactivity auto-disable applies to *scheduled* workflows in *public* repositories — the check itself is scope-agnostic and fails any non-active state.
- **CHECK:** `workflow-state` (reused kind) — one recorded forge query per workflow file matched by CI-01's glob: state `active` passes; any other state (in practice the `disabled_*` family, incl. GitHub's 60-day inactivity auto-disable) fails, naming the state (tree presence gate → forge query; `on_unreachable: skip`).
- **FIX:** `gh workflow enable <file>` (or the Actions UI); a commit or manual dispatch resets the inactivity clock.
- **SOURCE:** mechanism lift — baseline `rules/ops.json` OPS-07, citing GitHub Docs, *Disabling and enabling a workflow* — https://docs.github.com/en/actions/managing-workflow-runs-and-deployments/managing-workflow-runs/disabling-and-enabling-a-workflow (**note:** outside the four research angles; cited from baseline's own rule file, not invented).

### CI-06 — Deploy jobs depend on green build/test jobs
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** A deploy that can run against a red build ships the failure; in-workflow job ordering is the declared form of "each stage gates the next". Line-scan structural inference (no YAML parser), hence heuristic — and hence it stays a warning at every tier (see §Locked resolutions, R2: the skeleton's blocker-at-critical hardening was **not** locked).
- **CHECK:** `workflow-job-order` (new kind) — within one workflow file, the job block containing deploy signals (CI-03's patterns) also contains a `needs:` line referencing the job block(s) containing build/test signals (CI-02's patterns) (tree; heuristic).
- **FIX:** Add `needs: [build, test]` (the repo's actual job names) to the deploy job.
- **SOURCE:** "Progressively validate software changes across multiple environments, including development (alpha) and testing (beta) before deploying into production. ... Each non-production deployment serves as a gate, only allowing changes to progress to the next stage after they pass all validations. Early issue detection and isolation prevent propagation to later stages or production." — AWS, *DevOps Guidance [DL.ADS.1]* — https://docs.aws.amazon.com/wellarchitected/latest/devops-guidance/dl.ads.1-test-deployments-in-pre-production-environments.html (from `docs/archive/research/cicd-environments.md`, Claim 3).

## ENV — environment tiers

### ENV-01 — Environments are declared and agree across planes
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** The descriptor's tier list, the forge's environments, and the workflows' `environment:` keys are three declarations of one fact; drift between them is how "we have a staging" becomes untrue. Set agreement is deterministic.
- **CHECK:** `env-inventory` (new kind) — collect three sets: descriptor `environments[].name`, forge environments (`gh api repos/{owner}/{repo}/environments`), `environment:` values from workflow line-scans. PASS iff descriptor-set == forge-set == workflow-set; when the forge is unreachable, PASS iff descriptor-set == workflow-set, labeled `forge: not consulted`. Any symmetric difference between consulted sets FAILs, naming the differing tier(s) (tree + forge).
- **FIX:** Reconcile the three surfaces: declare each real environment in the descriptor, create missing forge environments, remove phantom ones.
- **SOURCE:** "Use multiple environments to experiment, develop, and test your workload. Use increasing levels of controls as environments approach production to gain confidence your workload operates as intended when deployed." — AWS, *Well-Architected Framework, OPS05-BP08* — https://docs.aws.amazon.com/wellarchitected/latest/framework/ops_dev_integ_multi_env.html · Also: "Environments are used to describe a general deployment target like `production`, `staging`, or `development`." — GitHub Docs, *Deployment environments* — https://docs.github.com/en/actions/concepts/workflows-and-actions/deployment-environments (from `docs/archive/research/cicd-environments.md`, Core claim C).

### ENV-02 — Every declared tier has an automated deploy path
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** A declared environment nothing deploys to is a diagram, not a tier. Per DORA, any version deployable to any environment on demand is the bar; a tier deployed by an external system the checker can't see is a documented `deviation`.
- **CHECK:** per descriptor `environments[].name`, a workflow references that tier (`environment: <name>` key or a per-tier workflow filename match) (tree; heuristic).
- **FIX:** Add a deploy job for the orphaned tier; externally-deployed tier → `pipeline jdg new --kind deviation --subject ENV-02 --reason "tier <name> deployed by <system>" --review-by <date>`.
- **SOURCE:** "Allow anyone with the necessary credentials to deploy any version of the artifact to any environment on demand in a fully automated fashion. If you have to create a ticket and wait for someone to prepare an environment, you don't have a fully automated deployment process." — DORA, *Capabilities: Deployment automation* — https://dora.dev/capabilities/deployment-automation/ (from `docs/archive/research/cicd-environments.md`, Claim 2).

### ENV-03 — The dev tier deploys on merge to the integration branch
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Pre-production tiers receive frequent automated deploys from the integration branch, and each passed tier gates the next — this is what catches integration failures before staging/prod. **Research-narrowed:** no primary source prescribes a literal *daily* deploy to a dev environment (DORA's daily cadence applies to builds/tests; AWS's to progressive gating), so this rule checks "on merge / frequent", never a daily quota, and stays team-tier warning.
- **CHECK:** line scan: the workflow deploying the descriptor's dev-tier environment triggers on `push` to the integration branch (`branching_model`-dependent: `trunk` → default branch; `dev-staging-main`/`gitflow` → the integration branch — descriptor `branches.integration`, falling back to `dev`/`develop` by model when unset) (tree; heuristic).
- **FIX:** Point the dev-deploy workflow's `on: push: branches:` at the integration branch.
- **SOURCE:** "Progressively validate software changes across multiple environments, including development (alpha) and testing (beta) before deploying into production. ... Each non-production deployment serves as a gate, only allowing changes to progress to the next stage after they pass all validations." — AWS, *DevOps Guidance [DL.ADS.1]* — https://docs.aws.amazon.com/wellarchitected/latest/devops-guidance/dl.ads.1-test-deployments-in-pre-production-environments.html · Cadence support (builds/tests, not deploys): "You should run your build process successfully at least once a day." — DORA, *Capabilities: Continuous integration* — https://dora.dev/capabilities/continuous-integration/ (from `docs/archive/research/cicd-environments.md`, Claim 3 — **NARROWED**, do not upgrade).

### ENV-04 — Production promotion is gated
- **Severity:** warning at team · **blocker at critical** (`severity_by_profile`) · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** For high-criticality systems, promotion into production requires explicit human approval — the forges ship exactly this mechanism and NIST supplies the compliance-grade why. Deterministic forge read, so blocker-at-critical is lawful.
- **CHECK:** `env-protection` (new kind) — read protection rules on the descriptor's prod-tier environment via `gh api repos/{owner}/{repo}/environments/<name>`; PASS when required reviewers (≥1) or an equivalent protection rule (wait timer + restricted branches) is configured (forge; `on_unreachable: skip` — offline, a labeled SKIP, never a silent PASS).
- **FIX:** Add required reviewers to the production environment (GitHub: environment → deployment protection rules → required reviewers; GitLab: protected environments + deployment approvals).
- **SOURCE:** "Deployment protection rules require specific conditions to pass before a job referencing the environment can proceed. Implement approved configuration-controlled changes to the system; Retain records of configuration-controlled changes to the system" — NIST SP 800-53 Rev. 5, CM-3 — https://nvlpubs.nist.gov/nistpubs/specialpublications/NIST.SP.800-53r5.pdf · Critical-tier practice precedent: "We decided to take a more conservative approach to deployments and deploy each change in stages. ### ENV-05 — Staging data realism is reviewed
- **Severity:** sign-off · **Profiles:** critical · **Certainty:** judgment · **applies_to:** `["service","app"]`
- **Rationale:** Pre-prod validation is only meaningful if staging resembles production, including data; data realism is inherently uncheckable at rest, so the entire rule is a dated human judgment — exactly as the skeleton predicted and research confirmed. **Context caveat:** the strongest quote is written for data pipelines (SRE Workbook ch.13); the principle generalizes, but the sign-off prompt quotes it with its context stated.
- **CHECK:** `signoff` — satisfied by an unexpired `kind: sign-off` judgment with subject `ENV-05` (ledger). The sign-off prompt asks: *staging data volume/shape reviewed against production within the review period; gaps named.*

### ENV-06 — A pre-prod validation stage runs before prod promotion
- **Severity:** warning · **Profiles:** critical · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Something automated must exercise staging before prod gets the change — smoke at minimum, load where traffic demands it. Presence is tree-checkable; *adequacy* for the failure-cost tier is the sign-off fallback. Stays a warning at critical (§Locked resolutions, R2).

- **SOURCE:** "Test release procedures in pre-production by using the same deployment configuration, security controls, steps, and procedures as in production." — AWS, *Well-Architected Framework, OPS06-BP02* — https://docs.aws.amazon.com/wellarchitected/latest/framework/ops_mit_deploy_risks_test_val_chg.html · Also: "Automate testing when deploying to production to simulate human and system interactions that verify the changes being deployed." — AWS, *Well-Architected Framework, OPS06-BP04* — https://docs.aws.amazon.com/wellarchitected/latest/framework/ops_mit_deploy_risks_auto_testing_and_rollback.html (from `docs/archive/research/cicd-environments.md`, Claims 5–6).

## IAC — infrastructure as code

### IAC-01 — An IaC tool is present
- **Severity:** blocker · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** At team scale, infrastructure that exists only in a console is unreproducible and unauditable — manual provisioning is the named failure source. **Research-narrowed:** sources say "best practice", not "prohibited" — so the rule requires the *mechanism* (blocker), and flags hand-applied changes (IAC-03) as a deviation-class warning, not a sin. A repo with genuinely no infrastructure (`docs`, `library`) SKIPs via `applies_to`.
- **CHECK:** `any-of` `any-file` over IaC markers: `*.tf`, `Pulumi.yaml`, `cdk.json`, `serverless.yml`, CloudFormation templates (`*.template`, `template.y*ml` with `AWSTemplateFormatVersion`), `*.bicep`, `pulumi.*.yaml` (tree).
- **FIX:** Define the infrastructure in an IaC tool (Terraform/CDK/Pulumi/Serverless/CFN/Bicep) and commit it alongside the app.
- **SOURCE:** "Infrastructure as code (IaC) provides a declarative approach to defining your infrastructure that creates version-controlled specifications you can review, test, and automate." — and: "**Eliminate manual configuration errors:** Manual infrastructure provisioning through cloud consoles leads to inconsistent configurations, missed steps, and deployment failures." — HashiCorp Well-Architected Framework — https://developer.hashicorp.com/well-architected-framework/define-and-automate-processes/define/as-code/infrastructure · Also: "You can commit your configurations to version control to safely collaborate on infrastructure." — HashiCorp Developer — https://developer.hashicorp.com/terraform/tutorials/aws-get-started/infrastructure-as-code (from `docs/archive/research/iac-drift.md`, Q1 — survives, narrowed).

### IAC-02 — IaC state is remote/shared; local state never committed
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Local state breaks team usage and committed state can leak secrets; HashiCorp states all three sub-claims (local fails teams, remote is the mechanism, never commit) verbatim. The team-tier floor is right: the cited complication is explicitly "when working with Terraform in a team".
- **CHECK:** `file-contains` for a backend block in `*.tf` (`backend "(s3|azurerm|gcs|remote|cloud|consul)"`) AND `any-file mode:absent` for `*.tfstate*` in tracked files AND `.gitignore` contains `*.tfstate` (tree). Non-Terraform IaC → the backend arm evaluates per-tool markers: Pulumi — a `backend:` key in `Pulumi.yaml` (or a `pulumi login --cloud-url` recorded in config); CDK — tool-managed state, SKIP with detail (`state managed by the CDK toolkit`); anything unresolved → SKIP with detail.
- **FIX:** Configure a remote backend (S3/HCP Terraform/etc.), delete any tracked state files, add `*.tfstate*` to `.gitignore`.
- **SOURCE:** "By default, Terraform stores state locally in a file named `terraform.tfstate`. When working with Terraform in a team, use of a local file makes Terraform usage complicated because each user must make sure they always have the latest state data before running Terraform and make sure that nobody else runs Terraform at the same time." — and: "With _remote_ state, Terraform writes the state data to a remote data store, which can then be shared between all members of a team." — HashiCorp Developer, *State: Remote Storage* — https://developer.hashicorp.com/terraform/language/state/remote · Also: "Avoid storing your state in a version control system or other storage solution that does not support Terraform state locking and secure access control, because doing so can result in data loss or exposure of secrets stored in the state file." — HashiCorp Developer, *State* — https://developer.hashicorp.com/terraform/language/state (from `docs/archive/research/iac-drift.md`, Q3 — survives as written).

### IAC-03 — IaC is applied by the pipeline, not by hand
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** IaC that humans apply from laptops reproduces the console problem with extra steps: untracked who/what/when, no consistent workflow. Manual application is the flagged deviation (research narrowing: "best practice", not "prohibited") — accepted consciously via the ledger, not silently.
- **CHECK:** line scan of workflow files for the IaC tool's apply invocation (`terraform apply`, `pulumi up`, `cdk deploy`, `serverless deploy`, `aws cloudformation deploy`) (tree; heuristic).
- **FIX:** Add an apply job to the pipeline (plan on PR, apply on merge); accepted manual application → `pipeline jdg new --kind deviation --subject IAC-03 --reason "..." --review-by <date>`.
- **SOURCE:** "Although best practice is to use Terraform for all infrastructure changes to ensure consistent workflows and change visibility, your organization may occasionally need to make manual changes." — HashiCorp Developer, *Use health assessments to detect infrastructure drift* — https://developer.hashicorp.com/terraform/tutorials/cloud/drift-detection · Also: "Use configuration management systems to make and track configuration changes. These systems reduce errors caused by manual processes and reduce the level of effort to deploy changes." ... "**Common anti-patterns:** … Someone has updated your security groups and your web servers are no longer accessible. Without knowledge of what was changed you spend significant time investigating the issue extending your time to recovery." — AWS, *Well-Architected Framework, OPS05-BP03* — https://docs.aws.amazon.com/wellarchitected/latest/framework/ops_dev_integ_conf_mgmt_sys.html (from `docs/archive/research/iac-drift.md`, Q1/Q7).

### IAC-04 — Config drift is detected on a cadence
- **Severity:** warning · **Profiles:** critical · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Drift is documented as *expected and unpreventable* — the failure is assuming it absent. - **CHECK:** `any-of`: (1) line scan — a scheduled workflow (`schedule:`/`cron` line) whose steps invoke the tool's plan/drift command (`terraform plan`, `pulumi preview --refresh`, `driftctl`) (tree; heuristic); (2) `signoff` — an unexpired judgment with subject `IAC-04` recording that a human reviews drift on a named cadence (drift-watch sign-off).
- **FIX:** Add a nightly cron workflow running `plan`/refresh-and-detect, failing (or filing) on diff; or record the drift-watch sign-off with a `review_by`.
- **SOURCE:** "Over time, your resources may change outside of the Terraform workflow. This can be due to service failures or degradation, certificate expirations, or manual modification by other users. Terraform cannot prevent these changes, but health assessments help you detect them quickly so you can resolve them." — HashiCorp Developer — https://developer.hashicorp.com/terraform/tutorials/cloud/drift-detection · The mechanism itself: "Implement drift detection as part of your CI/CD pipeline to catch configuration issues before they reach production. ### IAC-05 — Per-environment IaC configuration is separated
- **Severity:** warning · **Profiles:** critical · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Parity comes from replicating *one* definition across tiers with per-environment differences held in variables, not from three divergent hand-copies. **Research-narrowed:** 12-factor is application-config-shaped; the Terraform per-environment patterns close the infra gap — the rule scans for exactly those documented shapes.
- **CHECK:** line scan/glob for one of: named workspaces (`terraform workspace` invocations or `workspace` blocks), per-env variable files (`*.tfvars` under per-env paths, e.g. `envs/prod/`, `prod.tfvars`), per-env stack dirs (`stacks/`, `envs/`, `live/` containing per-tier configuration), or Pulumi per-stack files (`Pulumi.<stack>.yaml`) (tree; heuristic).
- **FIX:** Restructure to one definition + per-env tfvars/workspaces/stacks; a genuinely single-environment shop SKIPs via profile (team tier doesn't carry this rule).
- **SOURCE:** "There are also a variety of ways to handle multiple environments. The most common approaches are: … All environments use the same main branch, and environment differences are handled with Terraform variables. To protect production environments, wait to apply runs until their changes are verified in staging." — HashiCorp Developer, *Manage Terraform configurations* — https://developer.hashicorp.com/terraform/enterprise/workspaces/configurations · Also: "Some backends support multiple named workspaces, allowing multiple states to be associated with a single configuration. ### IAC-06 — No plaintext secrets in IaC files; tfvars never committed
- **Severity:** blocker · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Secrets in IaC source land in state and plan files, and in git history they are unrecoverable without rotation — a blocker is proportionate. The check is deliberately the *deterministic signature tier* only (private-key blocks, cloud key shapes) plus the tfvars git-ignore rule HashiCorp itself points at; softer entropy/assignment heuristics stay with baseline's SEC family, which this rule cross-references rather than restates (overlap control, SPEC.md §5). - **CHECK:** `any-of` (fail when ANY arm fires): (1) grep deterministic secret signatures over IaC globs (`**/*.tf`, `**/*.tfvars`, `**/Pulumi*.yaml`, `**/*.bicep`): `-----BEGIN [A-Z ]*PRIVATE KEY-----`, `AKIA[0-9A-Z]{16}` (AWS access key id shape); (2) tracked `*.tfvars` / `*.tfvars.json` files present in git index, or `.gitignore` missing `*.tfvars` when tfvars exist on disk (tree + history; deterministic).

## FLAG — feature flags (opt-in family)

**Activation (locked):** the family runs only when the descriptor sets `uses_feature_flags: true` **or** `records/flags/` exists — mirrors baseline's claims opt-in. A solo repo may legitimately skip the whole family; flags carry validation complexity and debt, so the standard never pushes them where failure cost doesn't justify the mechanism.

### FLAG-01 — A flag mechanism exists
- **Severity:** warning · **Profiles:** all (opt-in) · **Certainty:** heuristic · **applies_to:** `["service","app"]`
- **Rationale:** The defining purpose of release flags is decoupling deployment from release — a misbehaving feature becomes disableable without redeploying, which *shortens the recovery path*. **Honesty note (research gap 1):** no primary source *quantifies* kill-switch → MTTR; that conjunction is engineering inference joining two sourced ends, so the rationale claims the shorter path, not a measured number.
- **CHECK:** `any-of`: (1) grep flag-SDK imports over source globs (defaults: `src/**/*.{js,jsx,ts,tsx,py,go,rb}` plus the dependency manifests `package.json`, `requirements.txt`, `pyproject.toml`, `go.mod`, `Gemfile`; tunable via config `source_globs`) for `openfeature|@openfeature|launchdarkly|unleash|flagsmith|configcat|split\.io|@growthbook`; (2) `file-contains` on the descriptor for a `flags.store` field naming the mechanism (covers homegrown stores — the declaration is config, and FLAG-02/03 then police the registry it points at) (tree; heuristic).
- **FIX:** Adopt a flag mechanism (OpenFeature SDK + provider, or a declared config-driven store) and register flags per FLAG-02.
- **SOURCE:** "Release Toggles allow incomplete and un-tested codepaths to be shipped to production as latent code which may never be turned on. … Using Release Toggles in this way is the most common way to implement the Continuous Delivery principle of \"separating [feature] release from [code] deployment.\"" — Pete Hodgson, *Feature Toggles (aka Feature Flags)* — https://martinfowler.com/articles/feature-toggles.html · The kill switch: "The flag toggle button is a circuit breaker for any feature. The flag button lets you turn off a feature that's misbehaving without needing to touch any code or re-deploy your application." — LaunchDarkly Docs — https://docs.launchdarkly.com/home/flags/toggle/ · Vendor-neutral: "Feature flags allow application behavior to be altered without the deployment of new code." — OpenFeature Docs — https://openfeature.dev/docs/reference/intro/ (from `docs/archive/research/flags-rollback.md`, c-1 — survives as written).

### FLAG-02 — Every flag has a registry record with an owner
- **Severity:** warning · **Profiles:** team+ (opt-in) · **Certainty:** deterministic · **applies_to:** `["service","app"]`
- **Rationale:** One flag, one record (`records/flags/FLAG-NNNN.json`, schema `schema/record.flag.schema.json`) — patterned on baseline's per-claim records. The canonical article names exactly this metadata set; an unnamed-owner flag is how "who can turn this off?" becomes a 3am archaeology dig.
- **CHECK:** `flag-registry` (new kind, `assert: owner`) — every `records/flags/*.json` parses against the flag schema and has a non-empty `owner` (tree; deterministic). Empty-registry behavior: when the family is active (opt-in declared) but zero records exist, the rule WARNs with detail `flag registry empty` — the opt-in declares flags exist, so an empty registry is a finding, not a vacuous pass (mirrors baseline's zero-claims-record behavior).
- **FIX:** `cp templates/flag.json records/flags/FLAG-NNNN.json` per live flag; fill `name`, `owner`, `type`, `created`, `review_by`.
- **SOURCE:** "Some teams also opt to include additional metadata in their toggle configuration files such as a creation date, a primary developer contact, or even an expiration date for toggles which are intended to be short lived." — Pete Hodgson, *Feature Toggles* — https://martinfowler.com/articles/feature-toggles.html (from `docs/archive/research/flags-rollback.md`, §2.2 — the registry record's exact field set. **Source class:** canonical reference article — the toggle-metadata field set originates here; acceptable as primary citation for a registry-schema claim).

### FLAG-03 — Flags carry expiry; none past-due
- **Severity:** warning · **Profiles:** team+ (opt-in) · **Certainty:** deterministic · **applies_to:** `["service","app"]`
- **Rationale:** Flags are inventory with a carrying cost; stale flags are an operational hazard, verbatim from the sources. This check is the mechanical version of the cited "time bomb" practice. **Locked exemption:** permanent types (`kill-switch`, `permission` — per Fowler's long-lived ops toggles and Unleash's type table) need no near-term expiry; they carry long review horizons instead.
- **CHECK:** `flag-registry` (new kind, `assert: freshness`) — every registry record has `review_by` present unless `type ∈ {kill-switch, permission}` (those require `review_by` within config `permanent_flag_review_days`, default 365), and no record's `review_by` is before the run date (tree; deterministic date math against run date).
- **FIX:** Set/repair `review_by` per flag (defaults informed by Unleash's shipped lifetimes: release 40d, experiment 40d, operational 7d, sunset 90d, kill-switch/permission permanent-class); retire past-due flags or record `pipeline jdg new --kind risk-acceptance --subject FLAG-03 --reason "flag <name> kept because ..." --review-by <date>` with a tripwire.
- **SOURCE:** "Savvy teams view the Feature Toggles in their codebase as inventory which comes with a carrying cost and seek to keep that inventory as low as possible. In order to keep the number of feature flags manageable a team must be proactive in removing feature flags that are no longer needed. ... Other teams put \"expiration dates\" on their toggles. Some go as far as creating \"time bombs\" which will fail a test (or even refuse to start an application!) if a feature flag is still around after its expiration date." — Pete Hodgson, *Feature Toggles* — https://martinfowler.com/articles/feature-toggles.html · The hazard: "Feature flag technical debt accumulates when you don't manage or retire feature flags after their intended use. ... Additionally, the presence of stale or conflicting feature flags can lead to unexpected application behavior, increasing the risk of downtime and affecting overall stability." — and the precedent: "Unleash marks all flags as potentially stale automatically once they pass their expected lifetime. This gives you an indication of when to review and clean up a feature flag in code." — Unleash Docs — https://docs.getunleash.io/concepts/technical-debt (per-type lifetimes: https://docs.getunleash.io/concepts/feature-flags) (from `docs/archive/research/flags-rollback.md`, c-2 — survives as written, strongly).

## RB — rollback & recovery

### RB-01 — A rollback path is documented
- **Severity:** blocker · **Profiles:** all · **Certainty:** deterministic · **applies_to:** service · app · infra
- **Rationale:** The rollback plan must exist *before* the deploy that needs it — "it is better to have a well-thought out plan in place instead of trying to make one up when your service is on fire" is the strongest sentence in the corpus, and it holds at solo (the tier whose whole posture is fast recovery). Deploy-scoped twin of baseline OPS-05 (overlap control: this check scans for revert/redeploy steps, not a generic runbook). **Gated to deployable types** (matching CI-03) so a pure `docs`/`library` repo — including a no-descriptor repo that auto-detects as `docs` — is not red-flagged for lacking a deploy rollback (follow-up #9; honors §5.3's "a no-descriptor repo is not a red build").
- **CHECK:** `file-contains` over `RUNBOOK.md`, `runbooks/**/*.md`, `docs/runbooks/**/*.md`, `README.md`, `docs/*.md`: a rollback/revert section (pattern `(?=[\s\S]*(rollback|roll back|revert|redeploy))(?=[\s\S]*(deploy|release|version|previous))`) (tree; deterministic).
- **FIX:** Add a rollback section to the README or a runbook: how to revert/redeploy-previous for *this* repo's deploy mechanism (`templates/runbook-rollback.md`).
- **SOURCE:** "Before you plan to roll out a new binary or image to your service, you should ask yourself, \"What will I do if I discover a catastrophic / debilitating / annoying bug in this release?\" Not because it might happen, but because sooner or later it is going to happen and it is better to have a well-thought out plan in place instead of trying to make one up when your service is on fire." — Google Cloud Blog, *SRE at Google: Reliable releases and rollbacks — CRE life lessons* — https://cloud.google.com/blog/products/gcp/reliable-releases-and-rollbacks-cre-life-lessons · Also: "A \"Big Red Button\" is a unique but highly practical safety feature: it should kick off a simple, easy-to-trigger action that reverts whatever triggered the undesirable state to (ideally) shut down whatever's happening. … and it's important to identify what those big red buttons might be before you submit a potentially risky action." — Google SRE, *Lessons learned from twenty years of SRE* — https://sre.google/resources/practices-and-processes/twenty-years-of-sre-lessons-learned/ (from `docs/archive/research/flags-rollback.md`, c-3 — survives as written).

### RB-02 — Rollback is verified by drill
- **Severity:** sign-off · **Profiles:** critical · **Certainty:** judgment · **applies_to:** `["service","app","infra"]`
- **Rationale:** A documented-but-never-exercised rollback is the named anti-pattern; recovery capability is *demonstrated*, not asserted. Three independent primary sources converge (drill cadence, game days, test-before-emergency), so the sign-off-only design is deliberate: the checker can't observe a drill at rest, so a dated judgment with a tripwire is the honest mechanism.
- **CHECK:** `signoff` — an unexpired `kind: sign-off` judgment with subject `RB-02`, carrying drill evidence (date, what was rolled back, what broke) and `review_by` within config `rollback_drill_days` (default 90); recommended tripwire: `today gt <last-drill + rollback_drill_days>` shape via the ledger's fact ops (ledger).
- **FIX:** Run a rollback drill (roll back the last deploy in staging, time it, note traps), then `pipeline jdg new --kind sign-off --subject RB-02 --reason "drilled <date>: <what happened>" --review-by <date>`.
- **SOURCE:** "If you haven't rolled back in a few weeks, you should do a rollback \"just because\"; aim to find any traps with incompatible versions, broken automation/testing etc. If the rollback works, just roll forward again once you've checked out all your logs and monitoring." — Google Cloud Blog, *CRE life lessons* — https://cloud.google.com/blog/products/gcp/reliable-releases-and-rollbacks-cre-life-lessons · The anti-pattern: "Common anti-patterns: … You document your procedures, but your never exercise them." ### RB-03 — A mechanical fast-rollback path exists
- **Severity:** warning · **Profiles:** team+ · **Certainty:** heuristic · **applies_to:** `["service","app","infra"]`
- **Rationale:** "Easy to perform" is the mechanical property this check looks for at rest — a rollback that requires re-engineering the deploy under pressure is not a rollback path; "trusted to be low-risk" is RB-02's drill residue.
- **CHECK:** `any-of`: (1) the deploy workflow exposes a version/ref input (`workflow_dispatch` with a `version|ref|sha|tag` input line) enabling redeploy-previous; (2) the rollback doc (RB-01's surface) describes a revert-and-redeploy flow (`git revert` + redeploy) referencing the pipeline (tree; heuristic).
- **FIX:** Add a `workflow_dispatch` `version` input to the deploy workflow that deploys an arbitrary prior ref; or document the revert-and-redeploy flow precisely in the runbook.
- **SOURCE:** "At Google, our philosophy is that \"rollbacks are normal.\" When an error is found or reasonably suspected in a new release, the releasing team rolls back first and investigates the problem second. … Thus, for rollbacks to work, the implicit assumption is that they are: 1. easy to perform; and 2. trusted to be low-risk." — Google Cloud Blog, *CRE life lessons* — https://cloud.google.com/blog/products/gcp/reliable-releases-and-rollbacks-cre-life-lessons (from `docs/archive/research/flags-rollback.md`, §3.2).

### RB-04 — A recovery objective is declared and reviewed
- **Severity:** sign-off · **Profiles:** critical · **Certainty:** judgment · **applies_to:** `["service","app"]`
- **Rationale:** Recovery time is a first-class measured objective, and the SRE governance pattern — declare an objective, measure it, review the gap against actuals — is precisely this ledger entry. **Research-narrowed:** sources establish measure-and-review instead of *blanket* gates; they do not calibrate by failure-cost tier — that mapping stays the descriptor's judgment (SPEC.md §2). The declaration (descriptor `recovery_objective`) is machine-readable; whether it is *met* is dated human judgment.
- **CHECK:** two parts: (1) descriptor `recovery_objective` present and non-empty (tree; the declaration enables the rule) — absence yields WARN-level detail on the sign-off finding; (2) `signoff` — an unexpired `kind: sign-off` judgment with subject `RB-04` reviewing the objective against actual incidents/drills (ledger).
- **FIX:** Declare `recovery_objective` in the descriptor (e.g. "failed deployment recovery time < 30 min"), then review it against the incident/drill record each period and sign off with evidence. Vocabulary lock: DORA renamed MTTR → *failed deployment recovery time* in 2023 — use the current name.
- **SOURCE:** "Failed deployment recovery time: The time it takes to recover from a deployment that fails and requires immediate intervention." — DORA, *DORA's software delivery performance metrics* — https://dora.dev/guides/dora-metrics/ (rename note: "The metric historically known as \"mean time to recover (MTTR)\" or \"time to restore service\" was renamed and redefined as failed deployment recovery time." — https://dora.dev/insights/dora-metrics-history/) · The governance pattern: "It's both unrealistic and undesirable to insist that SLOs will be met 100% of the time: doing so can reduce the rate of innovation and deployment, require expensive, overly conservative solutions, or both. ## HOT — hotfix flow & incident response

### HOT-01 — A hotfix path is documented
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app","infra"]`
- **Rationale:** Critical production defects warrant an expedited path branching from production, and an official source states the process must be *defined and enforced* — which is what this check verifies. **Caveat locked into the check:** nvie's own 2020 reflection scopes GitFlow to versioned software ("If your team is doing continuous delivery of software, I would suggest to adopt a much simpler workflow (like GitHub flow)") — so the rule checks the hotfix *documentation* (branch off main, expedited pipeline, named approver), never mandates the full GitFlow branch set.
- **CHECK:** `file-contains` over `CONTRIBUTING.md`, `README.md`, `docs/*.md`, `runbooks/**/*.md`: hotfix section with all three elements — branch-from-production, expedited path, approver (pattern `(?=[\s\S]*hotfix)(?=[\s\S]*(branch|branched off|from main|from master|from prod))(?=[\s\S]*(approv|review|sign.?off))`) (tree; deterministic).
- **FIX:** Add a hotfix section (`templates/hotfix.md`): branch off the production tag (BR-06 guarantees one exists), which checks still run, who may approve the expedited merge.
- **SOURCE:** "Hotfix branches are very much like release branches in that they are also meant to prepare for a new production release, albeit unplanned. They arise from the necessity to act immediately upon an undesired state of a live production version." — Vincent Driessen (nvie), *A successful Git branching model* — https://nvie.com/posts/a-successful-git-branching-model/ · The definition mandate: "If you intend to lock a long-lived branch, it is critical to define your hotfix process and enforce it. If undefined and unenforced, every change becomes a hotfix." — GitLab Docs, *Branching strategies* — https://docs.gitlab.com/user/project/repository/branches/strategies/ (from `docs/archive/research/hotfix-incidents.md`, D1 — survives as written).

### HOT-02 — Hotfixes are back-merged into development branches
- **Severity:** warning · **Profiles:** critical · **Certainty:** heuristic · **applies_to:** `["service","app"]`
- **Rationale:** A hotfix that never reaches the development line silently regresses in the next release — the canonical source states the safeguard rationale verbatim. The history heuristic proves practice where hotfix history exists; a repo with zero hotfixes so far falls through to the sign-off arm rather than passing vacuously.
- **CHECK:** `any-of`: (1) `hotfix-backmerge` (new kind) — every `hotfix/*` branch tip reachable from remotes is an ancestor of the development branch (target: descriptor `branches.integration`, falling back to the model convention — `dev`/`develop`/default branch — when unset); returns not-evaluable when no `hotfix/*` refs exist (history; heuristic); (2) `signoff` — an unexpired judgment with subject `HOT-02` recording that back-merge is the agreed practice (the zero-hotfix-history fallback).
- **FIX:** Back-merge the named hotfix branches into the development line; no hotfix history yet → dated sign-off that back-merge is the agreed practice, with `review_by`.
- **SOURCE:** "When finished, the bugfix needs to be merged back into `master`, but also needs to be merged back into `develop`, in order to safeguard that the bugfix is included in the next release as well. This is completely similar to how release branches are finished." — Vincent Driessen (nvie), *A successful Git branching model* — https://nvie.com/posts/a-successful-git-branching-model/ · Same discipline, official: "At the same time, any hotfix branches are based off of the most recent release (`1.0`) of `main`, and merge back into `main` as release `1.1`. The `2.0` branch then pulls in the changes from release `1.1`, and incorporates them as part of the development of `2.0`." — GitLab Docs, *Branching strategies* — https://docs.gitlab.com/user/project/repository/branches/strategies/ (from `docs/archive/research/hotfix-incidents.md`, D2 — survives as written).

### HOT-03 — An incident response doc exists before the incident
- **Severity:** warning · **Profiles:** team+ · **Certainty:** deterministic · **applies_to:** `["service","app"]`

- **SOURCE:** "Prepare. ### HOT-04 — Break-glass events are recorded, owned, and expiring
